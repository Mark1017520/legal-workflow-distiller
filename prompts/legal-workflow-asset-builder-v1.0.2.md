# 法律工作流资产创建器 v1.0.2

请直接复制本文件到已经完成第一阶段审计的同一个 Codex 对话中使用。用户无需修改本提示词内容。

本提示词用于第二阶段：承接 `legal-workflow-distiller-v1.0.2.md` 的第一阶段审计输出，把已识别的候选法律工作流转化为可复用资产草案，例如 Skill、custom subagent、automation、template 或 checklist。

## 0. 本轮模式：ASSET_BUILD_CONTINUATION

你现在进入：

```text
ASSET_BUILD_CONTINUATION 资产创建接续模式
```

本提示词本身即表示用户希望进入第二阶段：

```text
开始创建资产
```

但是，这不等于授权你立即写入文件。你必须先基于上文第一阶段审计输出生成“拟创建方案”，并等待用户明确确认后，才可以创建、修改或写入任何文件。

如果用户只想先看草案，你应只在对话中输出草案，不得写文件。

## 1. 自动承接上文

请先从本对话上文中寻找最近一次由第一阶段提示词生成的审计输出，重点提取：

1. 可访问资料范围；
2. 现有资产覆盖矩阵；
3. 候选工作流列表；
4. 候选工作流评分表；
5. Top 3 资产化建议；
6. Skip 项；
7. 人工复核点；
8. 后续创建顺序。

如果找不到第一阶段审计输出，不要猜测。请停止，并要求用户粘贴第一阶段审计输出或重新运行第一阶段提示词。

如果找到了第一阶段审计输出，但用户没有额外指定候选工作流，则默认选择第一阶段 Top 1 推荐作为候选，并先输出“默认候选确认卡”。

如果用户在同一条消息里额外指定了候选工作流、资产类型、路径或写入权限，则优先使用用户指定内容。

## 2. 默认选择规则

当用户没有指定候选工作流时，按以下顺序自动选择：

1. 第一阶段明确标注的 Top 1 资产化建议；
2. 若无 Top 1，则选择评分最高且敏感度最低的候选工作流；
3. 若多个候选并列，则选择最适合沉淀为 checklist 或 template 的候选；
4. 若仍无法判断，则列出 2-3 个候选，让用户选择。

自动选择后，不要直接写文件。先输出：

```text
默认候选确认卡：
- 默认候选工作流：
- 选择理由：
- 原评分或排序依据：
- 建议资产类型：
- 建议目标路径：
- 是否发现相近已有资产：
- 需要用户确认的问题：
```

## 3. 默认资产类型和路径

如果第一阶段已经推荐资产类型，优先使用第一阶段推荐。

如果第一阶段没有明确推荐，则按以下规则判断：

- 结构稳定、可重复执行、适合 Codex 调用的工作流：优先做成 Skill；
- 判断风险高、必须人工主导的工作流：优先做成 checklist；
- 输出格式稳定、以文本交付为主的工作流：优先做成 template；
- 需要固定角色、边界和拒绝规则的工作流：可做成 custom subagent 草案；
- 涉及定时、监控、外部系统动作的工作流：仅输出 automation 方案，不得直接配置自动化。

默认路径规则：

```text
Skill: skills/<workflow-slug>/SKILL.md
Custom subagent: subagents/<workflow-slug>.md
Automation: automations/<workflow-slug>.md
Template: templates/<workflow-slug>.md
Checklist: checklists/<workflow-slug>.md
```

`<workflow-slug>` 应使用简短英文小写短横线命名。若候选工作流是中文名称，请翻译成可读的英文 slug。

如果目标路径已经存在，必须停止并询问用户是否扩展、覆盖或另存为新文件。

## 4. 创建规则

只创建用户明确确认的资产。

规则：

1. 每轮最多创建 1-3 个最小可用资产。
2. 除非用户明确批准，不得覆盖已有文件。
3. 不得写入客户名称、项目名称、合同正文、争议事实、受保密保护的法律意见、个人信息、凭证或商业秘密。
4. 将具体个案示例改写为中性占位符。
5. 必须包含人工复核点和停止条件。
6. 必要时写明维护负责人和更新触发条件。
7. 如果第一阶段审计发现已有资产有明显重叠，应优先提出“扩展已有资产”的方案，而不是重复创建。
8. 不得为了显得完整而虚构法规、事实、来源、历史案例或团队制度。

## 5. 不同资产类型的要求

### Skill

Skill 草案必须包含：

- 名称和简短说明；
- 适用场景；
- 不适用场景；
- 必需输入；
- 执行流程；
- 输出格式；
- 质量门槛；
- 升级或转人工触发条件；
- 人工复核点；
- 保密规则。

### Custom Subagent

Subagent 草案必须包含：

- 角色定位；
- 可以执行的任务；
- 必须拒绝或升级的任务；
- 输入边界；
- 输出边界；
- 复核清单；
- 失败模式。

### Automation

Automation 草案必须包含：

- 触发条件；
- 执行频率；
- 数据来源；
- 允许执行的动作；
- 禁止执行的动作；
- 通知路径；
- 人工批准节点；
- 日志和留存规则。

### Template

Template 草案必须包含：

- 使用目的；
- 必需事实；
- 占位符；
- 可选章节；
- 法律复核提示；
- 填写完成清单。

### Checklist

Checklist 草案必须包含：

- 适用范围；
- 前置条件；
- 按主题分组的检查项；
- 所需证据；
- 红旗问题；
- 关闭标准；
- 复核人签署或确认要求。

## 8. 停止条件

出现以下情况时，必须停止并询问用户：

- 找不到第一阶段审计输出；
- 已选工作流包含未脱敏的保密事实；
- 目标路径已经存在且用户未明确授权处理方式；
- 资产类型不清楚且无法从第一阶段推荐中推断；
- 候选工作流缺少评分、排序或推荐理由；
- 资产内容会固化具体法律意见，而不是沉淀工作流结构；
- 资产需要当前工作区之外的最新法律研究或法规检索；
- 用户要求把保密个案事实、合同原文、监管函件原文或个人信息写入通用资产。
