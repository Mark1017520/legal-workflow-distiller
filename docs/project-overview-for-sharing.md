# 项目分享说明

这是一份适合发布到小红书、公众号、团队知识库或 GitHub 项目介绍页的说明稿。它用更直接的方式解释本项目的定位、使用场景和安全边界。

## 一句话介绍

法律工作流蒸馏器是一个给 Codex / Claude Code / 本地工作区使用的提示词项目，用来从近期法律工作中识别可复用流程，并判断这些流程适合沉淀成 Skill、subagent、automation、template 还是 checklist。

## 为什么法律人需要它

很多法律工作看起来是一次性的，但背后其实有稳定结构：

- 先确认资料范围；
- 再识别法律问题；
- 再判断事实缺口；
- 再输出风险等级、修改建议和人工复核点；
- 最后决定是否可以标准化复用。

如果每次都从零开始写 prompt，很容易出现三个问题：

- prompt 越写越长，但复用性不稳定；
- 个案事实被误固化进通用流程；
- 团队不知道哪些工作适合 AI 辅助，哪些必须保持人工主导。

这个项目要解决的不是“让 AI 替代法律判断”，而是帮助法律团队把重复出现的法律工作拆解成可维护的工作流资产。

## 它和普通法律 Prompt 有什么不同

普通法律 prompt 往往直接要求 AI 输出答案，例如“帮我审合同”“帮我看风险”“帮我写合规意见”。

本项目不是直接做法律结论，而是先做只读审计：

- 当前工作区里有哪些材料；
- 这些材料体现了哪些重复法律任务；
- 哪些任务值得沉淀；
- 哪些任务风险太高，只适合作为 checklist；
- 哪些任务已经有资产覆盖，只需要扩展；
- 哪些任务不应该沉淀。

所以它更像法律运营工具，而不是单次法律问答工具。

## 适合什么场景

适合：

- 公司法务整理近期重复性工作；
- 隐私合规团队沉淀 PIA / DPIA / 数据出境判断流程；
- 产品法务整理产品上线审查流程；
- 合同团队沉淀合同初审、条款风险清单、谈判 fallback；
- 监管事务团队整理监管响应、问询回复、材料清单流程；
- 法律运营负责人规划团队 AI 工作流资产。

不适合：

- 直接生成最终法律意见；
- 直接处理未脱敏客户材料；
- 直接替代律师或法务负责人审批；
- 用公开模型处理受保密义务约束的文件；
- 把个案事实自动变成通用模板。

## 为什么适合 Codex / Claude Code / 本地工作区

这类本地工作区代理可以读取项目目录、文档、示例和已有资产。相比普通聊天窗口，它更适合做三件事：

- 看见资料结构，而不是只看一段粘贴文本；
- 识别已有 Skill、模板、清单和文档是否重复；
- 在用户确认后，把候选工作流进一步生成可维护的文件草案。

本项目默认只读运行。只有用户明确进入资产创建阶段后，才允许生成新文件。

## 使用后能沉淀什么资产

运行一次审计后，通常会得到：

- 候选工作流列表；
- 候选工作流评分表；
- Top 3 资产化建议；
- 已有资产覆盖矩阵；
- 不建议沉淀的 skip 项；
- 必须人工复核的高风险点；
- 后续创建顺序。

确认后，可以进入第二阶段，生成：

- Codex Skill 草案；
- custom subagent 规则草案；
- 自动化流程说明；
- 审查模板；
- checklist；
- 团队落地说明。

## 安全边界

本项目最重要的设计是 AUDIT_ONLY 只读模式。

在审计阶段，它只能观察和分析，不应该创建、修改、删除、移动或提交任何文件。这样可以降低两个风险：

- 避免未经确认就把敏感法律材料写入新资产；
- 避免 AI 误把个案判断当成团队通用规则。

使用前建议先完成：

```text
docs/pre-run-sanitization-checklist.md
```

不要直接放入客户真实名称、合同正文、监管函件原文、个人信息、API key、token、cookie、日志密钥、调查材料或其他受保密义务约束的内容。

## 推荐分享文案

我整理了一个开源提示词项目：法律工作流蒸馏器。

它不是让 AI 直接给法律意见，而是帮助法律、合规、隐私、产品法务团队从近期工作中识别可复用流程，并判断这些流程适合沉淀成 Skill、subagent、automation、template 还是 checklist。

项目默认是 AUDIT_ONLY 只读模式。也就是说，在你明确确认之前，它不会创建或修改任何文件，只会先输出候选工作流、评分、Top 3 资产化建议、人工复核点和后续创建顺序。

适合想把日常法律工作从“每次重新问 AI”升级为“可维护团队工作流资产”的人。
